
# 🚀 Internal Developer Platform (IDP): The MLOps Hub

This repository houses the declarative infrastructure and GitOps configurations for a modern **Internal Developer Platform (IDP)** tailored for Machine Learning Operations.

Our goal is to eliminate infrastructure toil by providing Data Scientists with a seamless, "one-click" experience. Through a **Backstage** developer portal, users can provision ephemeral, fully isolated ML workspaces (utilizing **AWS SageMaker**, **Bedrock**, and **vcluster**).

Behind the scenes, the platform relies on a "Lights-Out" automation sequence. **Terraform** provisions the immutable cloud foundation, while **ArgoCD** and **Crossplane** orchestrate the networking, security policies, and cloud resources in a strict, declarative GitOps loop.

---

## ⚡ Quick Start: Running a Demo

The platform uses a "Zero-Touch" bootstrap model. Once the initial pipeline is triggered, Terraform builds the cluster and hands control to ArgoCD, which deploys the entire platform across 6 automated deployment waves.

### **Prerequisites**

* An AWS Account and a GitHub Organization.
* GitHub OIDC trust established for your AWS Account.
* A GitHub Personal Access Token (PAT) with repository read access.

### **Step 1: Configure Global Variables**

We use a Single Source of Truth (SSOT) pattern. Open `global-variables.yaml` at the root of the repository and set your target deployment parameters:

```yaml
aws_region: "us-east-1"
account_id: "123456789012"
cluster_name: "platform-hub"
environment: "production"

```

### **Step 2: Bootstrap the Terraform State**

Run the initialization script to generate a secure, encrypted Amazon S3 backend for Terraform state storage (utilizing native conditional locking).

```bash
chmod +x ./mgmt_hub/scripts/init_scripts/root_module.sh
./mgmt_hub/scripts/init_scripts/root_module.sh

```

### **Step 3: Inject Secret Zero**

Add your GitHub PAT to your GitHub Repository Secrets as `PLATFORM_GITHUB_TOKEN`. This allows the Terraform pipeline to securely pass repository credentials to the ArgoCD bootstrap controller.

### **Step 4: Deploy the Platform**

Commit your changes and push to the `main` branch.

1. **GitHub Actions** will run `terraform apply` to build the VPC and EKS Cluster.
2. Terraform installs **ArgoCD** and applies the `platform-applicationset.yaml`.
3. ArgoCD takes over, executing a strict 6-Wave rollout (Network -> API -> Database -> Secrets -> Security -> FinOps).

Within 15 minutes, the platform is fully operational and the Backstage UI is accessible.

## 🏗️ Architecture Details

The system is built on a strict **Hub-and-Spoke** architecture, segregated into three distinct engineering layers to ensure blast-radius containment and separation of concerns.

### **Layer 1: The Foundation (Infrastructure as Code)**

Managed entirely by Terraform, this layer represents the immutable AWS hardware and network perimeter.

* **Compute:** Amazon EKS (v1.31+) utilizing dedicated Node Groups (System vs. Tenant workloads).
* **Networking:** 3-Tier Private VPC architecture.
* **Identity:** Workload Identity via the EKS Pod Identity Agent (Zero-secret AWS authentication).

### **Layer 2: The Engine Room (GitOps Platform)**

The core platform services deployed automatically by ArgoCD onto the Hub cluster's System nodes.

* **Networking:** **Cilium** deployed in strict eBPF mode with WireGuard mTLS for encrypted pod-to-pod traffic.
* **Cloud API:** **Crossplane** handles the provisioning of AWS resources (RDS, S3) directly from Kubernetes manifests.
* **Secrets Management:** A self-hosted **Infisical** operator, dynamically retrieving database credentials generated by Crossplane.
* **Security & FinOps:** **Kyverno** and Native Validating Admission Policies (VAP) enforce read-only filesystems and mandatory cost-center labeling, integrated with **OpenCost**.

### **Layer 3: The Developer Interface**

The user-facing portal and tenant sandboxes.

* **Portal:** **Backstage** utilizing Dynamic Plugins (no custom Docker builds required).
* **Workspaces:** **vcluster** provides cryptographically isolated, ephemeral Kubernetes control planes for individual ML teams, seamlessly integrated with Okta/GitHub OIDC for RBAC.

## 📂 Repository Structure

The codebase is strictly organized to separate Terraform infrastructure from Kubernetes GitOps configurations.

```text
.
├── global-variables.yaml            # Single Source of Truth for Region/Account
├── Backstage/                       # Developer Portal Source Code & Templates
│   └── template/ml_workspace/
├── mgmt_hub/                        # Layer 1: Cloud Foundation
│   ├── Docs/                        # Technical Architecture Decisions (Compute, IAM)
│   ├── scripts/                     # S3 Backend and Bootstrap utilities
│   └── terraform/                   # AWS VPC, EKS, and ArgoCD Bootstrap
└── platform_interface/              # Layer 2 & 3: GitOps Manifests
    └── kubernetes/
        ├── bootstrap/               # The ArgoCD "God Object" ApplicationSet
        ├── platform/                # Core System Tools (Wave 1-6)
        │   ├── argocd/
        │   ├── backstage/
        │   ├── cilium/
        │   ├── crossplane/
        │   ├── finops/              # OpenCost
        │   ├── globals/             # ConfigMap Generator for SSOT
        │   ├── infisical/
        │   ├── observability/       # OTel, Loki, Tempo, Prometheus
        │   ├── rds/                 # Crossplane Database Manifests
        │   └── security/            # Kyverno & VAP Guardrails
        └── tenants/                 # ML Workspaces generated by Backstage

```
